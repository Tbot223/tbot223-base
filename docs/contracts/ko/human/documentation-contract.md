# Documentation Contract

> Python docstring이 아닌 저장소 Markdown 문서에 적용하는 문서 작성 계약.

## 1. 목표

- README, 가이드, 레퍼런스, 릴리스 노트, 계약 문서의 스타일을 통일한다.
- 각 문서가 어떤 역할인지 문서 상단에서 바로 드러나게 한다.
- README, Guide, Reference는 런타임 기준 정보를, Contract는 문서 정책 revision 정보를 드러내게 한다.
- en/ko 문서의 파일 구조는 맞추되, 문장까지 기계적으로 같게 만들지는 않는다.
- 문장 자체보다 문서 프레임을 통일한다.
- GitHub Flavored Markdown을 기본 작성 형식으로 삼는다.

## 2. 적용 범위

이 계약은 다음에 적용한다.

- 루트 README와 언어별 README.
- `docs/en`, `docs/ko` 아래 장문 문서.
- `docs/contracts` 아래 계약 문서.
- 저장소의 Markdown 문서.

이 계약은 다음에는 적용하지 않는다.

- `TODO.md`.
- 코드 주석.
- 소스 파일이나 예제 스크립트.
- 별도 docstring 계약을 따르는 Python docstring.

## 3. 계약 강도

이 문서는 중간 강도의 계약이다.

- 모든 문서는 하나의 주된 문서 유형을 MUST 가진다.
- 모든 문서는 상단 근처에서 범위나 목적을 MUST 드러낸다.
- 모든 문서는 해당 유형의 기본형을 SHOULD 따른다.
- 정확한 헤더 이름이나 문장 모양까지 동일할 필요는 없다.

## 4. 메타와 공통 규칙

- README, Guide, Reference 문서는 상단 메타 블록에 런타임 기준 정보를 MUST 명시한다.
- Contract 문서는 상단 메타 블록에 `Contract revision` 또는 `Last updated`를 MUST 명시한다.
- Contract 문서는 패키지 버전의 부속 문서가 아니라 문서 정책 revision을 기록하는 문서로 본다.
- Release Notes는 버전 히스토리 자체가 기준 정보 역할을 하므로 런타임 기준이나 revision 표기를 별도로 요구하지 않는다.
- 상단 메타 블록의 기본 순서는 `언어 전환 링크 -> 기준 정보 -> 제목`으로 둔다.
- 대응 언어 문서가 없으면 `기준 정보 -> 제목`으로 시작할 수 있다.
- 첫 10줄 안쪽에서 범위, 요약, 또는 독자 의도를 드러낸다.
- 한 문서 안에는 하나의 주된 모드를 유지한다. README, 가이드, 릴리스 노트, 레퍼런스가 뒤섞일 경우에는 섹션 경계를 분명히 둔다.
- 헤더 깊이는 얕게 유지한다. 더 깊은 중첩은 탐색성이 실제로 좋아질 때만 쓴다.
- 설명은 문단으로, 나열은 리스트로, 고정 구조 비교는 표로 쓴다.
- 기본적으로 GitHub Flavored Markdown을 기준으로 작성하고, GitHub 렌더링이 올바르게 보이도록 우선한다.
- fenced code block에는 가능한 한 언어 태그를 붙인다.
- 내부 링크는 상대 경로를 유지한다.
- 대응하는 다른 언어 문서가 있다면 상단 근처에 언어 전환 링크를 둔다.
- 런타임 기준 정보는 자연어 문장, 짧은 배너, blockquote 등 문서에 맞는 형식으로 둘 수 있다.
- 정확한 릴리스 버전을 검증할 수 없으면 `current dev branch 기준`, `main branch HEAD 기준`처럼 검증 가능한 임시 기준을 쓰고, 추정 버전을 만들지 않는다.
- Contract 문서의 revision 정보는 저장소에서 확인 가능한 날짜나 명시적 revision 라벨만 사용하고, 패키지 버전을 추정해 쓰지 않는다.
- 새로 쓰거나 수정하는 문서는 현재 저장소의 `.markdownlint.jsonc` 기준에서 피할 수 있는 경고를 SHOULD 추가하지 않는다.
- 기존 문서를 수정할 때는 현재 저장소의 `.markdownlint.jsonc` 기준에서 비용이 낮은 경고를 가능하면 함께 줄인다.
- 저장소 설정에서 구조 규칙이 이미 비활성화되어 있으면, 일반 markdownlint 기본 규칙에 맞추려고 문서 구조를 억지로 바꾸지 않는다.
- 초안 메모, TODO 성격의 문장, 잡담성 표현은 마무리 전에 제거한다.

## 5. 이중 언어 정책

- en/ko 문서가 모두 존재하는 경우 경로와 파일명은 MUST 맞춘다.
- 섹션 순서와 문장 구성은 언어별로 달라도 된다.
- 다루는 정보 범위는 SHOULD 비슷하게 유지한다.
- 언어 전환 링크, 고유명사, 불가피한 기술 용어를 제외하고 본문을 양쪽 언어로 섞지 않는다.

## 6. 문서 유형

### README

진입점과 방향 제시가 필요한 문서에 사용한다.

필수 요소:

- 런타임 기준 표기.
- 제목.
- 짧은 요약.
- 대상 독자, 목적, 또는 프로젝트 성격 설명.
- 더 깊은 문서로 가는 링크.

권장 순서:

```text
언어 링크(선택) -> 기준 정보 -> 제목 -> 요약 -> 대상/목적 -> 핵심 섹션 -> 추가 문서/링크
```

문체:

- 설명형.
- 직접적.
- 과장 없는 안내형.

### Guide

마이그레이션, 설정, 워크스루, 작업 절차 문서에 사용한다.

필수 요소:

- 런타임 기준 표기.
- 제목.
- 짧은 요약.
- 필요하면 시작 상태나 전제 조건.
- 순서가 보이는 흐름 또는 단계 구분.
- 필요하면 기대 결과, 검증 지점, 다음 행동.

권장 순서:

```text
언어 링크(선택) -> 기준 정보 -> 제목 -> 요약 -> 전제 조건/상황 -> 단계 또는 진행 섹션 -> 검증/다음 행동
```

문체:

- 절차형.
- 행동 지향형.
- 전환과 결과를 분명히 설명하는 방식.

### Reference

API 개요, 구조 설명, 조회형 문서에 사용한다.

필수 요소:

- 런타임 기준 표기.
- 제목.
- 범위 설명.
- 탐색 가능한 구조.
- 필요할 때 표, 고정 포맷 블록, 반복 가능한 항목 구조.

권장 순서:

```text
언어 링크(선택) -> 기준 정보 -> 제목 -> 범위 -> 탐색/목차(선택) -> 기능 블록 -> 필요 시 Note/Example
```

문체:

- 압축형.
- 정확한 설명 중심.
- 불필요한 서사 최소화.

### Release Notes

버전 히스토리와 변경 보고 문서에 사용한다.

필수 요소:

- 제목.
- 버전/날짜 단위 구분.
- 내용에 맞는 변경 카테고리.

권장 순서:

```text
제목 -> 버전 블록 -> 변경 카테고리
```

문체:

- 사실 중심.
- 짧고 단정하게.
- 튜토리얼처럼 흐르지 않게.

### Contract

저장소 규약, 작성 계약, 표준 문서에 사용한다.

필수 요소:

- contract revision 또는 last updated 표기.
- 목적.
- 범위.
- 규칙.
- 체크리스트, 워크플로, 또는 동반 문서 안내.

권장 순서:

```text
언어 링크(선택) -> 기준 정보 -> 제목 -> 목적 -> 범위 -> 규칙 -> 워크플로/체크리스트 -> 동반 문서
```

문체:

- 규범형.
- 명확한 구분 중심.
- 필요할 때 MUST, SHOULD, MAY를 사용한다.

## 7. 일관성 규칙

- 문서 상단과 첫 주요 헤더에서 문서 유형이 드러나야 한다.
- README 문체를 릴리스 노트에 그대로 쓰지 않는다.
- 릴리스 노트의 묶음 방식을 가이드 문서에 그대로 가져오지 않는다.
- 레퍼런스 문서를 긴 에세이처럼 바꾸지 않는다. 혼합 모드가 필요하면 섹션 경계를 분명히 둔다.
- 문서 안에서 용어는 일관되게 유지한다. 용어가 바뀌면 변경 사실을 설명한다.
- 이중 언어 쌍 중 하나를 수정하면, 다른 언어 파일도 구조 정렬과 기준 정보 표기 정합성이 필요한지 같이 본다.

## 8. 최종 체크리스트

- 문서 유형이 상단에서 드러나는가?
- README, Guide, Reference라면 런타임 기준 정보가 상단 메타에 있는가?
- Contract라면 contract revision 또는 last updated가 상단 메타에 있는가?
- 기준 정보가 저장소에서 검증 가능한 사실에 기대고 있는가?
- 시작 블록이 문서의 용도를 설명하는가?
- 구조가 문서 유형에 맞는가?
- GitHub에서 Markdown 렌더링이 자연스럽게 보이는가?
- 내부 링크가 상대 경로이며 가능한 범위에서 같은 언어 트리를 가리키는가?
- 코드 블록에 언어 태그가 있는가?
- 현재 저장소의 `.markdownlint.jsonc` 기준에서 새 경고를 추가하지 않았는가?
- TODO 성격 문장이나 초안 메모를 제거했는가?
- 대응 언어 파일이 있다면 경로가 여전히 맞는가?
